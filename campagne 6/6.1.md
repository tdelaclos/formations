# Pourquoi ce chapitre existe

Depuis le début de ce manuel, nous avons progressivement construit une infrastructure de plus en plus robuste.

Nous avons appris à protéger :

- les comptes utilisateurs ;
- les permissions UNIX ;
- les ACL ;
- Firewalld ;
- systemd ;
- le sandboxing ;
- les Linux Capabilities.

À ce stade, notre serveur est déjà très bien protégé.

Et pourtant...

Un problème fondamental subsiste encore.

Imaginons que Sentinel possède une vulnérabilité Remote Code Execution.

Un attaquant parvient à exécuter :

```python
os.system(...)
```

Que peut-il faire ?

La réponse est simple.

Tout ce que Sentinel est autorisé à faire.

Autrement dit :

- lire ses fichiers ;
- écrire dans ses répertoires ;
- ouvrir des connexions réseau ;
- communiquer avec d'autres processus.

Jusqu'ici, rien d'anormal.

Mais réfléchissons plus loin.

Sentinel tourne sous l'utilisateur :

```text
sentinel
```

Cet utilisateur possède bien certains droits.

Et c'est précisément le problème.

Le noyau Linux traditionnel ne sait répondre qu'à une seule question :

> **L'utilisateur possède-t-il les permissions nécessaires ?**

Il ne se demande jamais :

> **Cette action est-elle cohérente avec le rôle de cette application ?**

Cette nuance va entièrement justifier l'existence de SELinux.

# Théorie détaillée

## Revenons au modèle UNIX

Depuis les premiers chapitres,

nous avons abondamment manipulé les permissions.

Par exemple.

```text
-rwxr-x---
```

ou

```bash
chmod
```

ou encore

```bash
chown
```

Toutes ces commandes reposent sur le même modèle.

Le contrôle d'accès discrétionnaire.

En anglais :

```text
Discretionary Access Control

DAC
```

---

# Pourquoi "Discretionary" ?

Le terme est rarement expliqué.

Il est pourtant très important.

Discrétionnaire signifie que :

> **Le propriétaire d'une ressource décide lui-même qui pourra y accéder.**

Prenons un exemple.

```text
/home/alice/document.txt
```

Alice est propriétaire.

Elle peut décider :

```bash
chmod 777
```

ou

```bash
chmod 600
```

Elle contrôle donc les permissions.

Le noyau applique simplement sa décision.

C'est exactement le principe du DAC.

---

# Le fonctionnement du DAC

Lorsqu'un processus tente d'accéder à un fichier,

Linux vérifie essentiellement :

```text
Qui est l'utilisateur ?

↓

Qui est le groupe ?

↓

Quelles permissions possède le fichier ?

↓

Autoriser ?

↓

Refuser ?
```

Le raisonnement est relativement simple.

Il fonctionne très bien depuis plus de quarante ans.

Mais...

Il possède une faiblesse majeure.

---

# Le problème des applications compromises

Prenons Sentinel.

Il fonctionne sous :

```text
Utilisateur :

sentinel
```

Le répertoire :

```text
/var/lib/sentinel
```

lui appartient.

Il peut donc :

```text
Lire

↓

Modifier

↓

Créer

↓

Supprimer
```

Jusqu'ici,

tout est normal.

Supposons maintenant qu'un attaquant compromette Sentinel.

Que voit le noyau ?

```text
Processus

↓

Utilisateur sentinel

↓

Permissions valides

↓

Autorisé
```

Le noyau ne sait absolument pas distinguer :

```text
Le vrai Sentinel
```

de

```text
Un attaquant

ayant compromis Sentinel.
```

Pour lui,

les deux sont exactement le même processus.

Voilà la limite fondamentale du DAC.

---

# Une analogie

Imaginons une entreprise.

Chaque employé possède un badge.

Le système actuel fonctionne ainsi.

```text
Badge valide ?

↓

Oui

↓

Accès autorisé.
```

Supposons maintenant qu'un voleur dérobe le badge.

Le contrôle d'accès répondra exactement la même chose.

```text
Badge valide ?

↓

Oui.
```

Il est incapable de distinguer :

- le véritable employé ;
- le voleur.

Le DAC présente exactement cette faiblesse.

Il fait confiance au processus parce qu'il possède une identité valide.

Il ne vérifie jamais si son comportement reste cohérent.

---

# L'héritage des permissions

Une autre conséquence apparaît immédiatement.

Imaginons que Sentinel exécute :

```python
subprocess.run(...)
```

Le nouveau processus hérite naturellement :

- de l'utilisateur ;
- du groupe ;
- des permissions.

Schématiquement.

```text
Sentinel

↓

subprocess

↓

Même utilisateur

↓

Mêmes permissions
```

Encore une fois,

le noyau considère que tout est normal.

Pourtant,

ce nouveau processus pourrait être entièrement contrôlé par un attaquant.

---

# Le problème n'est pas root

Beaucoup pensent que SELinux sert principalement à empêcher les attaques root.

C'est faux.

Le problème apparaît bien avant.

Même un simple utilisateur applicatif peut représenter un risque important.

Prenons :

```text
Utilisateur :

apache
```

ou

```text
postgres
```

ou

```text
sentinel
```

Ces comptes possèdent souvent :

- des certificats ;
- des clés privées ;
- des fichiers de configuration ;
- des secrets d'API ;
- des connexions réseau.

Compromettre l'un d'eux suffit parfois à compromettre une grande partie d'un système d'information.

Le problème n'est donc pas uniquement root.

Le problème est :

> **Toute identité dont les privilèges peuvent être détournés.**

---

# La naissance d'une nouvelle idée

À la fin des années 1990,

la NSA travaille sur une question fondamentale.

> Peut-on construire un système dans lequel même un processus compromis resterait fortement limité ?

La réponse donnera naissance à :

```text
Security-Enhanced Linux

SELinux
```

L'idée paraît simple.

Le système ne doit plus seulement vérifier :

```text
Qui êtes-vous ?
```

Il doit également vérifier :

```text
Que cherchez-vous à faire ?

Est-ce cohérent avec votre rôle ?
```

Cette différence est énorme.

Elle va complètement changer la manière dont Linux contrôle les accès.

---

# Changer complètement de philosophie

Avec le modèle historique,

la décision ressemble à ceci.

```text
Utilisateur

↓

Permissions

↓

Autorisé
```

SELinux introduit une nouvelle étape.

```text
Utilisateur

↓

Permissions UNIX

↓

Politique SELinux

↓

Autorisé ?
```

Autrement dit,

les permissions UNIX ne disparaissent pas.

Elles deviennent simplement **une première condition**.

Une deuxième décision est ensuite prise.

Et cette deuxième décision est indépendante du propriétaire du fichier.

---

# Une conséquence majeure

Supposons que le fichier possède :

```text
777
```

En DAC,

cela signifie pratiquement :

```text
Tout le monde

↓

Accès autorisé.
```

Avec SELinux,

la réponse peut devenir :

```text
Permissions UNIX

↓

Oui

↓

Politique SELinux

↓

NON

↓

Accès refusé.
```

C'est probablement la première idée que le lecteur doit retenir.

Les permissions UNIX ne suffisent plus à prédire un accès.

Elles ne sont plus que l'une des conditions nécessaires.

---
# Le Mandatory Access Control (MAC)

Pour résoudre les limites du DAC,

SELinux introduit un modèle complètement différent.

Son nom est :

```text
Mandatory Access Control

MAC
```

Le mot important est :

```text
Mandatory
```

Contrairement au DAC,

les règles ne sont plus décidées par le propriétaire du fichier.

Elles sont imposées par le système lui-même.

Autrement dit,

même si Alice possède un fichier,

elle ne peut plus décider librement de tous les accès.

Le système conserve le dernier mot.

---

# DAC contre MAC

Comparons les deux approches.

## DAC

```text
Propriétaire

↓

Définit les permissions

↓

Le noyau applique
```

---

## MAC

```text
Administrateur sécurité

↓

Définit une politique globale

↓

Le noyau applique

↓

Même le propriétaire

ne peut pas l'ignorer
```

Le changement est considérable.

Le contrôle d'accès devient une politique de sécurité,

et non plus une simple conséquence des permissions d'un fichier.

---

# Une analogie

Reprenons notre exemple du badge.

Avec le DAC.

```text
Badge valide ?

↓

Oui

↓

Vous entrez.
```

Avec le MAC.

```text
Badge valide ?

↓

Oui

↓

Êtes-vous autorisé

à entrer dans CETTE salle ?

↓

Oui / Non
```

Même si le badge est parfaitement valide,

certaines salles restent interdites.

Le système ne fait plus uniquement confiance à votre identité.

Il tient également compte de votre rôle.

---

# Une autre analogie : l'hôpital

Imaginons un hôpital.

Un médecin possède un badge.

Une infirmière possède également un badge.

Les deux peuvent accéder au bâtiment.

Mais...

Peuvent-ils accéder :

- au coffre contenant les stupéfiants ?
- à la salle des serveurs ?
- au laboratoire biologique ?

Évidemment non.

Pourtant,

leurs badges sont valides.

Le contrôle d'accès ne dépend plus uniquement de l'identité.

Il dépend aussi :

- du rôle ;
- du contexte ;
- de la politique de sécurité.

C'est exactement ce que fait SELinux.

---

# Pourquoi Linux ne pouvait pas faire cela avant ?

Parce que les permissions UNIX ne contiennent que trois informations.

```text
Utilisateur

Groupe

Autres
```

Et trois types de droits.

```text
Lecture

Écriture

Exécution
```

Le noyau ignore totalement :

- quel logiciel effectue l'accès ;
- quel est son rôle ;
- pourquoi il ouvre ce fichier.

Pour Linux traditionnel,

un processus Apache compromis et Apache légitime sont identiques.

Ils possèdent :

- le même UID ;
- le même GID ;
- les mêmes permissions.

Le noyau ne peut donc pas faire la différence.

---

# Ce que SELinux ajoute

SELinux ajoute une nouvelle information.

Chaque objet possède désormais une identité de sécurité.

Par exemple :

```text
Fichier

↓

Contexte SELinux
```

Le processus possède également :

```text
Processus

↓

Contexte SELinux
```

La décision devient alors beaucoup plus riche.

```text
Processus

↓

Contexte

↓

Objet

↓

Contexte

↓

Politique

↓

Décision
```

Nous venons d'introduire un mot extrêmement important.

Le contexte.

Il constituera l'objet du prochain chapitre.

---

# Premier exemple avec Sentinel

Prenons un exemple volontairement simplifié.

Sentinel lit sa configuration.

```text
/opt/sentinel/conf/config.yml
```

Les permissions UNIX disent :

```text
Lecture autorisée.
```

Jusqu'ici,

rien de nouveau.

Mais SELinux ajoute une seconde vérification.

```text
Le processus Sentinel

est-il autorisé

à lire

ce type de fichier ?
```

Deux situations deviennent possibles.

---

## Cas n°1

```text
Permissions UNIX

↓

Oui

SELinux

↓

Oui

↓

Accès autorisé
```

---

## Cas n°2

```text
Permissions UNIX

↓

Oui

SELinux

↓

NON

↓

Permission denied
```

Le même utilisateur.

Le même fichier.

Les mêmes permissions.

Et pourtant,

la décision finale change.

C'est précisément ce qui déroute souvent les administrateurs découvrant SELinux.

---

# Pourquoi cela surprend autant ?

Imaginons la situation suivante.

Vous exécutez :

```bash
ls -l
```

Résultat.

```text
-rw-r--r--
```

Tout semble correct.

Le propriétaire est bon.

Le groupe est bon.

Les permissions sont bonnes.

Vous tentez alors :

```bash
cat fichier.txt
```

Et obtenez :

```text
Permission denied
```

Le premier réflexe est généralement :

> Les permissions UNIX sont incorrectes.

En réalité,

elles peuvent être parfaitement correctes.

C'est SELinux qui refuse l'accès.

Cette distinction est fondamentale.

Une grande partie des erreurs de diagnostic provient justement de cette confusion.

---

# SELinux ne remplace pas les permissions UNIX

C'est probablement l'idée la plus importante du chapitre.

Beaucoup d'administrateurs imaginent ceci.

```text
SELinux

↓

Remplace

chmod
```

C'est faux.

Le noyau applique les deux mécanismes.

Schématiquement.

```text
          Requête

              │

              ▼

      Permissions UNIX

              │

      Refus ?────────────► FIN

              │

             Oui

              ▼

      Politique SELinux

              │

      Refus ?────────────► FIN

              │

             Oui

              ▼

      Accès autorisé
```

Autrement dit,

pour qu'un accès soit autorisé,

**les deux contrôles doivent être satisfaits**.

Il ne suffit plus d'avoir les bonnes permissions UNIX.

Il faut également respecter la politique de sécurité.

---

# Pourquoi Red Hat a choisi SELinux

Au début des années 2000,

plusieurs mécanismes de contrôle d'accès obligatoire apparaissent sous Linux.

Parmi eux :

- SELinux ;
- AppArmor ;
- TOMOYO ;
- Smack.

Red Hat fait rapidement le choix de SELinux.

Pourquoi ?

Parce qu'il présente plusieurs caractéristiques particulièrement adaptées aux environnements professionnels :

- très grande granularité des règles ;
- séparation claire entre les politiques et les applications ;
- intégration profonde avec le noyau Linux ;
- modèle extensible ;
- performances compatibles avec les serveurs de production.

Aujourd'hui,

AlmaLinux,

RHEL,

CentOS Stream,

Fedora

utilisent tous SELinux comme mécanisme MAC principal.

Autrement dit,

lorsqu'un ingénieur maîtrise SELinux,

il maîtrise le modèle de sécurité utilisé par une très grande partie des infrastructures Linux professionnelles.

---

# Une nouvelle façon de raisonner

À partir de maintenant,

chaque fois qu'un accès sera refusé,

nous ne poserons plus une seule question.

Nous en poserons deux.

Première question.

```text
Les permissions UNIX

autorisent-elles l'accès ?
```

Deuxième question.

```text
La politique SELinux

autorise-t-elle cette opération ?
```

Ce double raisonnement deviendra progressivement un réflexe.

C'est probablement le plus grand changement intellectuel apporté par cette campagne.

# 💎 Le point d'expertise

## SELinux ne protège pas les fichiers. Il protège les interactions.

C'est probablement l'erreur de compréhension la plus fréquente.

Beaucoup d'administrateurs pensent que SELinux ajoute simplement des permissions supplémentaires aux fichiers.

Ce n'est pas du tout son fonctionnement.

Prenons un fichier.

```text
/etc/shadow
```

Avec les permissions UNIX, la question est simple.

```text
Le processus possède-t-il

les droits de lecture ?
```

SELinux pose une question complètement différente.

```text
Ce processus

a-t-il le droit,

selon son rôle,

de lire CE type de fichier ?
```

Autrement dit,

SELinux ne protège pas uniquement des objets.

Il contrôle les **relations** entre des objets.

Cette nuance est fondamentale.

---

## SELinux n'accorde pratiquement jamais d'autorisations

Une autre idée fausse consiste à penser que SELinux « donne des droits ».

En réalité,

SELinux fonctionne principalement comme un mécanisme de restriction.

Visualisons la séquence complète.

```text
Permissions UNIX

↓

Accès possible

↓

SELinux

↓

Accès éventuellement refusé
```

SELinux intervient donc généralement **après** les permissions UNIX.

Il retire des possibilités.

Il en ajoute très rarement.

Cette manière de raisonner est essentielle lorsqu'un diagnostic devient complexe.

---

## Le véritable objectif de SELinux

Imaginons deux serveurs.

### Premier serveur

```text
Apache compromis

↓

Lecture

de

/etc/shadow

↓

Possible
```

---

### Deuxième serveur

```text
Apache compromis

↓

Lecture

de

/etc/shadow

↓

Refus SELinux
```

Dans les deux cas,

Apache possède exactement la même vulnérabilité.

SELinux ne corrige donc absolument pas le bug.

En revanche,

il limite considérablement les conséquences de cette compromission.

Encore une fois,

nous retrouvons le principe de défense en profondeur.

Une vulnérabilité n'implique plus automatiquement une compromission complète.

---

## SELinux protège contre les erreurs humaines

Prenons maintenant un cas beaucoup plus courant.

Un administrateur exécute par erreur :

```bash
chmod 777 /etc/sentinel
```

Les permissions UNIX deviennent extrêmement permissives.

Sans SELinux,

le risque augmente immédiatement.

Avec SELinux,

la politique peut continuer à interdire certains accès.

Autrement dit,

SELinux protège parfois autant contre les erreurs d'administration que contre les attaques.

C'est l'une des raisons pour lesquelles il est si apprécié dans les grandes infrastructures.

---

# 🧠 Comment pense un architecte ?

Un architecte ne cherche jamais à protéger uniquement des fichiers.

Il cherche à protéger des **flux d'information**.

Prenons Sentinel.

Les flux réellement nécessaires sont les suivants.

```text
Sentinel

↓

Lecture

de sa configuration
```

---

```text
Sentinel

↓

Lecture

de ses certificats
```

---

```text
Sentinel

↓

Écriture

de sa base locale
```

---

```text
Sentinel

↓

Connexion TLS

vers FreeIPA
```

En revanche,

de nombreux autres flux n'ont aucune raison d'exister.

Par exemple.

```text
Sentinel

↓

Lecture

des HOME utilisateurs
```

---

```text
Sentinel

↓

Modification

de /usr/bin
```

---

```text
Sentinel

↓

Lecture

des bases PostgreSQL
```

L'architecte construit donc une politique qui autorise uniquement les flux réellement utiles.

Tout le reste est interdit.

## La sécurité devient déclarative

Avant SELinux,

la réflexion ressemble à ceci.

```text
Qui possède ce fichier ?
```

Avec SELinux,

la question devient.

```text
Quelles interactions

sont légitimes

dans cette architecture ?
```

Le changement est profond.

On ne protège plus des objets.

On décrit les comportements attendus du système.

---

# ⚔️ Comment pense un attaquant ?

Lorsqu'un attaquant obtient une exécution de code,

il commence presque toujours par explorer son environnement.

Par exemple.

```bash
find /

cat /etc/passwd

cat /etc/shadow

find / -name "*.pem"

ls /home
```

Dans un système ne reposant que sur le DAC,

ces commandes réussiront dès que les permissions UNIX le permettent.

Avec SELinux,

la situation change.

Même si le processus possède les permissions UNIX nécessaires,

la politique peut bloquer chacune de ces opérations.

Pour l'attaquant,

cela change complètement la stratégie.

Il ne lui suffit plus d'obtenir une exécution de code.

Il doit également comprendre,

puis contourner,

la politique SELinux.

Cela augmente considérablement la difficulté d'une attaque.

---

## La reconnaissance devient plus difficile

La première phase d'une compromission consiste presque toujours à comprendre l'environnement.

Quels services tournent ?

Quels fichiers sont présents ?

Quels certificats existent ?

Quels secrets sont accessibles ?

SELinux réduit fortement cette visibilité.

L'attaquant travaille avec beaucoup moins d'informations.

Or,

une attaque menée dans un environnement mal connu est généralement beaucoup moins efficace.

Cette perte de visibilité constitue un avantage majeur souvent sous-estimé.

# 🏢 En entreprise

Contrairement à une idée reçue,

SELinux est rarement activé "pour faire plaisir à Red Hat".

Il répond à des exigences très concrètes.

Prenons une entreprise exploitant plusieurs centaines de serveurs.

Chaque serveur héberge :

- des services Web ;
- des bases de données ;
- des API ;
- des outils d'administration ;
- des agents de supervision.

Même si tous ces services utilisent des utilisateurs dédiés,

ils restent exposés au même problème.

Une compromission permet d'agir avec tous les privilèges du compte compromis.

Pour limiter ce risque,

les équipes sécurité cherchent à compartimenter les applications.

SELinux devient alors un outil d'isolation extrêmement efficace.

---

## Une politique commune

Dans une infrastructure professionnelle,

les administrateurs n'écrivent généralement pas une politique spécifique pour chaque serveur.

Ils s'appuient d'abord sur la politique standard fournie par la distribution.

Celle-ci protège déjà :

- Apache ;
- Nginx ;
- OpenSSH ;
- Firewalld ;
- Chronyd ;
- Podman ;
- FreeIPA ;
- MariaDB ;
- PostgreSQL ;
- de nombreux autres services.

Les applications développées en interne,

comme Sentinel,

viennent ensuite compléter cette politique.

L'objectif est toujours le même :

intégrer une nouvelle application sans affaiblir la sécurité globale du système.

---

## Pourquoi les grandes entreprises refusent de désactiver SELinux

Un scénario revient régulièrement.

Une application ne fonctionne pas.

L'administrateur exécute :

```bash
setenforce 0
```

L'application fonctionne immédiatement.

Il conclut alors :

> « SELinux pose problème. »

Dans une entreprise mature,

ce raisonnement est considéré comme un signal d'alerte.

Pourquoi ?

Parce qu'il signifie que :

- la politique n'a pas été comprise ;
- l'application ne respecte peut-être pas les conventions du système ;
- le problème est masqué au lieu d'être résolu.

Une organisation mature préfère presque toujours :

- comprendre le refus ;
- corriger le contexte ;
- adapter éventuellement la politique ;

plutôt que supprimer une couche entière de sécurité.

# 📚 Culture technique

SELinux est souvent présenté comme un projet de la NSA.

C'est historiquement exact,

mais cette affirmation est parfois mal interprétée.

Au milieu des années 1990,

la NSA cherchait un moyen de renforcer la sécurité des systèmes UNIX utilisés dans des environnements sensibles.

Plutôt que de conserver cette technologie en interne,

elle a choisi de publier son travail sous licence libre.

Le projet est rapidement repris par la communauté Linux.

Puis,

Red Hat décide de l'intégrer profondément dans sa distribution.

Depuis,

SELinux est maintenu conjointement par :

- Red Hat ;
- la communauté Open Source ;
- de nombreux industriels.

Il ne s'agit donc plus aujourd'hui d'un projet expérimental.

C'est l'un des composants les plus éprouvés du noyau Linux.

## Pourquoi AppArmor existe également

Le lecteur rencontrera probablement un autre nom :

```text
AppArmor
```

AppArmor poursuit le même objectif général :

mettre en œuvre un contrôle d'accès obligatoire.

La philosophie est cependant différente.

De manière très simplifiée :

### AppArmor

Les règles sont principalement construites autour des chemins.

```text
/usr/bin/sentinel

↓

peut accéder

↓

/etc/sentinel/*
```

---

### SELinux

Les règles sont construites autour des types de sécurité.

```text
Processus

↓

Type

↓

Objet

↓

Type

↓

Politique
```

Les deux approches sont valides.

Red Hat,

AlmaLinux,

Fedora,

CentOS Stream

ont fait le choix de SELinux.

Ubuntu privilégie traditionnellement AppArmor.

Le présent ouvrage se concentre naturellement sur SELinux,

car il constitue aujourd'hui la référence sur les systèmes de la famille Red Hat.

---

# ⚠️ Piège classique

## Désactiver SELinux au premier refus

C'est probablement l'erreur la plus répandue.

Le scénario est presque toujours le même.

```bash
Permission denied
```

L'administrateur pense immédiatement :

> SELinux est trop compliqué.

Quelques secondes plus tard.

```bash
setenforce 0
```

Tout fonctionne.

Le problème semble résolu.

En réalité,

aucun problème n'a été résolu.

Une couche entière de sécurité vient simplement d'être supprimée.

C'est comparable à retirer les freins d'une voiture parce qu'ils empêchent d'accélérer.

Le rôle de cette campagne sera précisément d'apprendre à diagnostiquer ces refus,

afin de ne jamais avoir besoin de désactiver SELinux.

---

## Croire que SELinux "bloque tout"

Autre idée très répandue.

> « SELinux empêche les applications de fonctionner. »

En réalité,

des milliers de serveurs AlmaLinux et RHEL exécutent quotidiennement :

- Apache ;
- PostgreSQL ;
- FreeIPA ;
- Podman ;
- OpenSSH ;

avec SELinux activé,

sans aucune difficulté.

Le véritable problème apparaît presque toujours lorsque :

- une application est installée en dehors des conventions de la distribution ;
- des fichiers possèdent un mauvais contexte ;
- ou qu'une nouvelle fonctionnalité n'a pas été prise en compte dans la politique.

Autrement dit,

SELinux n'est généralement pas la cause du problème.

Il est celui qui le révèle.

# Laboratoire AlmaLinux / Kali

## Objectif

Prendre conscience des limites du contrôle d'accès discrétionnaire (DAC) et comprendre pourquoi un mécanisme MAC est nécessaire.

---

## Étape 1 — Observer les permissions UNIX

Créer un répertoire de test.

```bash
mkdir /tmp/dac-demo

touch /tmp/dac-demo/secret.txt

chmod 644 /tmp/dac-demo/secret.txt
```

Observer :

```bash
ls -l
```

Identifier précisément ce que les permissions autorisent.

---

## Étape 2 — Changer uniquement les permissions

Modifier les droits.

```bash
chmod 777 /tmp/dac-demo/secret.txt
```

Constater qu'un simple changement de permissions modifie complètement les accès.

Cette expérience rappelle que le DAC repose uniquement sur l'identité et les permissions.

---

## Étape 3 — Identifier les limites

Lister plusieurs ressources sensibles utilisées par Sentinel.

Par exemple :

- certificats TLS ;
- base locale ;
- journaux ;
- configuration.

Pour chacune,

répondre à la question suivante :

> Les permissions UNIX suffisent-elles réellement à protéger cette ressource si Sentinel est compromis ?

Argumenter chaque réponse.

---

## Étape 4 — Préparer la suite

Vérifier que SELinux est actif.

```bash
getenforce
```

Puis :

```bash
sestatus
```

Sans chercher encore à interpréter les informations.

L'objectif est simplement de constater que SELinux est déjà présent sur AlmaLinux avant même toute configuration.

# Mission d'ingénieur

Votre entreprise développe plusieurs applications métiers.

À la suite d'un audit de sécurité,

un scénario est retenu.

> Une vulnérabilité RCE est découverte dans l'une des applications.

Vous devez démontrer,

sans encore utiliser la moindre commande SELinux,

pourquoi les mécanismes étudiés jusqu'à présent (DAC, utilisateur dédié, permissions UNIX, sandbox systemd, Linux Capabilities)

ne suffisent pas à empêcher toutes les conséquences de cette compromission.

Votre rapport devra ensuite expliquer en quoi un mécanisme de contrôle d'accès obligatoire apporte une couche de protection supplémentaire.

L'objectif n'est pas de proposer une solution technique.

Il est de convaincre la direction de l'intérêt d'introduire SELinux dans l'infrastructure.

---

# Impact sur Sentinel

Jusqu'à présent,

Sentinel reposait principalement sur :

- un utilisateur dédié ;
- Firewalld ;
- systemd ;
- le sandboxing ;
- les Linux Capabilities.

À partir de cette campagne,

une nouvelle dimension apparaît.

Sentinel ne sera plus seulement limité par ses permissions UNIX.

Il sera également contraint par une politique de sécurité indépendante de son identité.

Même si une vulnérabilité permet à un attaquant d'exécuter du code dans le processus,

celui-ci devra encore respecter les règles imposées par SELinux.

Cette couche supplémentaire marque une évolution majeure de notre architecture de défense en profondeur.

---

# Ce qu'il faut retenir

- Les permissions UNIX mettent en œuvre un contrôle d'accès discrétionnaire (DAC).
- Le DAC repose sur l'identité du processus et les permissions des objets.
- Un processus compromis conserve les mêmes permissions que le processus légitime.
- SELinux introduit un contrôle d'accès obligatoire (MAC).
- Le propriétaire d'un fichier ne décide plus seul des autorisations.
- Les permissions UNIX restent nécessaires, mais elles ne suffisent plus.
- SELinux ajoute une seconde décision de sécurité indépendante des permissions classiques.
- Comprendre **pourquoi** SELinux existe est indispensable avant d'apprendre ses commandes.
